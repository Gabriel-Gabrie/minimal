🔄 Firestore Sync Test - Manual Verification Guide

📋 Overview

This guide walks you through manually verifying that the new data model fields (masterSections, masterSectionOrder, _dataVersion, and budgetMonths) are properly syncing to Firestore.

⚠️ Prerequisites:
  • You must have a Firebase account configured for this app
  • Access to Firebase Console (console.firebase.google.com)
  • Know your Firebase project ID
  • App must be running (http://localhost:8000)

🧪 Manual Testing Procedure

Step 1: Sign In to Firebase
  1. Open the app at http://localhost:8000
  2. Click the Settings icon (gear icon in top-right)
  3. Sign in with your Firebase account (Google, Apple, or Email)
  4. Wait for sign-in to complete
  5. Open browser DevTools (F12) → Console tab
Expected: Console should show "Signed in as [your email]" or similar message.
Step 2: Trigger a Data Change

Make any change that triggers saveData():

  • Option A: Add a new transaction
  • Option B: Create/edit a budget item
  • Option C: Add/edit a wallet account
  • Option D: Use browser console to manually trigger saveData:
// In browser console: saveData(); console.log('saveData() called manually');
Step 3: Wait for Debounced Firestore Write

The Firestore write is debounced by 1.5 seconds. Wait at least 2 seconds after the change.

// Optional: Check in console after 2 seconds: setTimeout(() => console.log('Firestore write should be complete now'), 2000);
Note: If the app has sync status indicators, watch for "Syncing..." → "Synced" status change.
Step 4: Check Firestore Console
  1. Open Firebase Console
  2. Select your project
  3. Navigate to Firestore Database in the left sidebar
  4. Find the users collection
  5. Click on your user document (document ID = your Firebase UID)
  6. Expand the data field
Step 5: Verify New Fields Are Present

In the Firestore document, verify the following fields exist inside the data object:

Field Name Expected Type Expected Content
masterSections Object (map) Section names as keys, arrays of category names as values
Example: { "Income": ["Salary", "Freelance"], "Bills": [...] }
masterSectionOrder Array Array of section names defining display order
Example: ["Income", "Bills", "Groceries", "Transport", ...]
_dataVersion Number Should be 1 (indicates migration completed)
budgetMonths Object (map) Month keys (YYYY-MM) as keys, each containing:
- activeSections (object)
- sectionOrder (array)
- budgets (nested object)
Step 6: Verify budgetMonths Structure

Expand the budgetMonths field and check one month (e.g., "2026-03"):

{ "budgetMonths": { "2026-03": { "activeSections": { "Income": ["Salary", "Freelance"], "Bills": ["Rent", "Utilities"] }, "sectionOrder": ["Income", "Bills"], "budgets": { "Income": { "Salary": { "Paycheque": { "amount": 5000 } } }, "Bills": { "Rent": { "Apartment": { "amount": 1500 } } } } } } }
✅ Expected Structure:
  • activeSections: Object mapping section names to arrays of category names
  • sectionOrder: Array of section names
  • budgets: Nested object (section → category → item → {amount})

🔄 Testing Firestore → App Sync (Bonus Test)

To verify that data can be read FROM Firestore back into the app:

Test 1: Clear Local Storage and Reload
  1. Open browser DevTools → Application → Local Storage
  2. Delete all localStorage items (or just the "budgetAppData" key)
  3. Reload the page (F5)
  4. App should automatically load data from Firestore
  5. Check console for "Loaded from Firestore" or similar message
// Verify in console after reload: console.log('masterSections:', masterSections); console.log('masterSectionOrder:', masterSectionOrder); console.log('_dataVersion:', _dataVersion); console.log('budgetMonths:', budgetMonths);
✅ Expected: All new fields are populated from Firestore and app functions normally.

❌ Troubleshooting

Issue: New fields not appearing in Firestore

  • Check: Are you signed in? Console should show your user email.
  • Check: Did you wait 2+ seconds? Firestore write is debounced by 1.5 seconds.
  • Check: Console errors? Look for "Firestore write:" warnings.
  • Check: Firestore rules? Ensure /users/{uid} is writable by authenticated user.

Issue: Fields exist but structure is wrong

  • Check: Did migration run? Console should show "Budget data migration to v1 complete".
  • Check: _dataVersion = 1? If 0, migration didn't complete.
  • Check: budgetMonths structure? Each month should have activeSections, sectionOrder, budgets.

Issue: App doesn't load data from Firestore

  • Check: Console for "Firestore read failed" warnings.
  • Check: Firestore rules allow read access to /users/{uid}.
  • Check: Network tab - is Firestore request completing?
